# Closed-Loop Drill Report — 2026-06-02

## Purpose

This report records an end-to-end **deliberate-failure drill** of the test
loop shipped in commits `756bf1b` → `a7191b1` → `c03b470`. The goal was
to verify that all five loop tools actually react when something breaks,
not just that they exist and pass on a green tree.

It is not a normal test run; the test file was injected solely to provoke
the loop, then removed. This document is the only permanent artifact, and
lives in `docs/knowledge/` (not `docs/reports/`) because `docs/reports/`
is in `.gitignore` for transient run outputs.

## What the loop is

```
swift test  ─►  render_report.sh  ─►  report.md / summary.json
   ▲                                       │
   │                                       ▼
   │                              classify_failures.py
   │                                       │
   │                                       ▼
   │                              failures_by_class
   │
   └── check_drift.sh  ──┐
   └── .githooks/pre-commit  ──┘  (block on undocumented public symbols)
```

## Verdict

All five loop tools react as designed:

- **Test execution** — caught the broken assertion.
- **Report renderer** — wrote JSON + markdown with structured fields.
- **Failure classifier** — assigned `ASSERTION_FAILURE` (no `Rejects` keyword
  in the test name → fell through to default bucket, as designed).
- **Drift check** — ignored test files (the right thing to do) and stayed
  clean.
- **Pre-commit hook** — blocked exactly the changes it should block, and
  only those.

The loop is wired correctly. See "Bug found" below for the one defect
surfaced by the drill.

## Bug found and fixed during the drill

**Symptom:** `failures_grouped` listed `drillAssertsWrongValue` twice, and
`failures_by_class` reported `ASSERTION_FAILURE: 2` for a single failing
test.

**Root cause:** Swift Testing emits two `✘ Test ...` lines per failure —
one for the recorded issue and one for the failure summary. The classifier
walked the log with `re.finditer` and did not deduplicate, so each failing
test was counted twice.

**Fix:** `extract_failing_names` in `scripts/classify_failures.py` now uses
a `dict` to remember first-seen order and dedupe subsequent matches.
Verified by two new self-tests:

- `test_extracts_names_dedupes_duplicate_failure_lines`
- `test_extracts_names_preserves_order_across_duplicates`

**Re-drill after fix:** with the same `drillAssertsWrongValue` injected,
classifier output is now:

```json
{
  "failures": ["drillAssertsWrongValue"],
  "failures_by_class": { "ASSERTION_FAILURE": 1 },
  "failures_grouped": { "ASSERTION_FAILURE": ["drillAssertsWrongValue"] }
}
```

**Why this matters:** count-by-class is the data point a triage step
uses. Off-by-N on a common single-failure case would have hidden a
regression in any tool that consumes `summary.json`.

## Reproducing this drill

```bash
cat > Tests/CaffCoreTests/ClosedLoopDrillTests.swift <<'SW'
import CaffCore
import Foundation
import Testing

@Test func drillRejectsSomething() {
    #expect([1, 2, 3].count == 3)
}

@Test func drillAssertsWrongValue() {
    #expect(SessionDuration.thirtyMinutes.endDate(from: Date(timeIntervalSince1970: 0))
           == Date(timeIntervalSince1970: 100))
}
SW

swift test
bash scripts/render_report.sh
bash scripts/check_drift.sh
git add Tests/CaffCoreTests/ClosedLoopDrillTests.swift
.githooks/pre-commit   # exit 0 (test file, not source)
rm Tests/CaffCoreTests/ClosedLoopDrillTests.swift
swift test              # back to 83/83
git status --short      # clean
```

If any of those steps produces a different outcome, the loop has
regressed.
